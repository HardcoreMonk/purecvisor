# ADR-0007: security_group 6 RPC를 S6 RBAC 시나리오에 흡수한다

날짜: 2026-04-10
상태: accepted

## 맥락
코드 감사(2026-04-10)에서 security_group 관련 6개 RPC가 어떤 시나리오에도
매핑되지 않음:

- security_group.create / delete / list / rule.add / rule.remove
- vm.security_group.set

이들은 nftables 기반 네트워크 보안 그룹(인그레스/이그레스 규칙)으로,
VM/컨테이너에 방화벽 정책을 적용하는 기능이다. §6 "멀티테넌트 권한 분리"
시나리오(S6)는 현재 RBAC(인증/권한)만 다루지만, 보안 그룹은 *네트워크
수준의 접근 제어*로 S6의 자연스러운 확장이다.

별도 시나리오(S17)로 만들지 않는 이유:
- 보안 그룹의 CRUD는 ADMIN 역할만 허용 — S6 RBAC과 동일 권한 모델
- 감사 로그 기록도 S6 인프라를 그대로 사용
- 규모가 작아(6 RPC) 독립 시나리오로 만들면 문서 비용만 증가

## 결정
S6을 "멀티테넌트 권한 분리 + 네트워크 보안 그룹"으로 확장:

1. DESIGN.md S6에 "네트워크 보안 그룹" 실패 모드·통과 조건 추가
2. §5 매핑표 S6 행에 security_group.* 추가
3. 통합 테스트: security_group CRUD 라운드트립 + VIEWER 거부 확인

## 결과
- 좋음: 고아 6건 → 자산 전환, S6의 보안 범위가 더 완전
- 나쁨: S6이 약간 비대해짐 (인증 + 네트워크 정책)
- 포기한 것: 별도 S17 "네트워크 보안 정책" 시나리오의 명확함

## 하지 않기로 한 것
- WAF/DPI/IPS 수준 정책. security_group은 L3/L4 필터링만.
- 보안 그룹의 클러스터 간 자동 동기화. 노드별 독립 적용.
