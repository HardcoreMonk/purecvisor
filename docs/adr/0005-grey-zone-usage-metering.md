# ADR-0005: 회색지대 ~52 RPC는 6개월간 사용 메트릭을 수집한 후 결정한다

날짜: 2026-04-10
상태: accepted

## 맥락
코드 감사(2026-04-10)에서 ~250 RPC 중 약 108건이 DESIGN.md 시나리오에
매핑되지 않음. ADR-0002~0004로 OVN(18) + DPDK/SR-IOV(14) + GPU/iSCSI/
Federation(10) = 42건을 격리 처리. 남은 ~66건 중:

- 명백한 비범위 (네트워크 고급 14, 스토리지 고급 11) → 별도 ADR 후보
- **회색지대 ~52건**: agent.*, ai.*, jobs.*, overlay.*, device.*, plugin.*,
  template.*(S12로 일부 흡수됨), security_group.*, audit.*, hot_reload.*

회색지대의 특징:
- 어떤 것은 자주 쓰이지만 시나리오에 명시되지 않음 (audit.*, hot_reload.*)
- 어떤 것은 거의 안 쓰이지만 §6 Won't Do와 직접 충돌하지는 않음 (ai.*)
- 어떤 것은 다른 RPC의 *내부 의존성*이라 직접 호출 빈도가 낮음 (jobs.*,
  device.*)

회고 교훈: "한 번에 정리"는 안티패턴. 정량 데이터 없이 회색지대를 결정
하면 또 다른 사후 패치 사이클이 시작된다.

## 결정
6개월(2026-10-10까지) 사용 메트릭을 수집한 후 카테고리별로 ADR을 추가
한다. 새 코드는 작성하지 않는다 — 이미 있는 audit log를 활용:

1. **데이터 소스**: `pcv_audit.db` (SQLite WAL, 30일 retention) +
   `audit.log`. 모든 변경 RPC가 method 이름과 함께 기록되고 있음.
2. **수집 스크립트**: `scripts/rpc_usage_report.sh` 신규 — audit DB를
   읽어 method별 호출 횟수, 마지막 호출 시각, 호출 사용자 수를 카테고리별
   집계 (작성은 본 ADR 범위 외, 스크립트만 정의)
3. **월 1회 리포트**: 매월 1일 cron으로 리포트 생성, `docs/usage/YYYY-MM.md`
   누적
4. **분류 임계값** (6개월 누적):
   - 호출 0건 → **삭제 후보** (다음 ADR 대상)
   - 호출 1~10건 / 단일 사용자 → **deprecated 후보** (3개월 추가 관찰)
   - 호출 11건 이상 / 복수 사용자 → **시나리오 추가 대상**
     (DESIGN.md S14+ 신설)
5. 6개월 후 일괄이 아닌 카테고리별 ADR 작성 (안티패턴 회피)

추가 RPC 격리/삭제는 본 ADR 기간(6개월) 동안 정지. 단, 명백한 보안 결함이
발견된 경우는 예외.

## 결과
- 좋음: 정량 데이터 기반 결정 — 추측 제거
- 좋음: 새 코드 작성 0 (audit 인프라 재사용)
- 좋음: 6개월간 추가 ADR 부담 없음 → 운영에 집중
- 좋음: 진짜 사용되는 회색지대는 시나리오로 흡수되어 부채 → 자산 전환
- 나쁨: 6개월 동안 부채 ~52건 그대로 유지
- 나쁨: audit log가 *조회* RPC는 기록 안 함 (변경 RPC만) → 일부
  카테고리는 데이터 부족 가능성. 그 경우 별도 telemetry counter 검토
  (본 ADR 범위 외)
- 포기한 것: "직감으로 정리하기"의 속도. 회고의 교훈을 우선.

## 하지 않기로 한 것
- 6개월 안에 회색지대를 추가 격리. 데이터가 없는 결정은 또 다른 사후
  패치를 만든다.
- 호출 0건 RPC를 즉시 삭제. 6개월 윈도우는 분기 보고/연 1회 사용 패턴까지
  포착하기 위함.
- 사용 메트릭을 Prometheus에 추가 노출. 그것은 카디널리티 폭증(메서드명
  라벨) 위험이 있고 N3(관측가능성) NFR과 별개 문제. audit DB로 충분.
- 리포트를 자동 의사결정에 연결(자동 삭제 PR 등). 결정은 사람이, 데이터는
  도구가.
