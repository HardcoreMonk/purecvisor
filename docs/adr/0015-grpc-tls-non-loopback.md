# ADR-0015: gRPC 비루프백 바인딩 시 TLS 강제

날짜: 2026-04-10
상태: accepted

## 맥락
gRPC 서버(grpc_server.c)는 내부 통신 경로로 남아 있다.
기본 bind_addr는 127.0.0.1(로컬 전용)이며, daemon.conf에서 0.0.0.0으로
변경할 수 있다. 인증은 `[grpc] auth_token` 설정의 16바이트 prefix 비교.

문제: 토큰이 TCP 소켓에 평문으로 전송된다. 127.0.0.1 바인딩이면
로컬 프로세스만 접근하므로 안전하지만, 0.0.0.0 바인딩 시 네트워크에서
토큰이 스니핑될 수 있다.

현재 부분 구현: 0.0.0.0 + 토큰 미설정 조합은 시작 거부(2026-04-09 패치).
그러나 0.0.0.0 + 토큰 설정 + TLS 미설정 조합은 허용됨.

코드 리뷰(2026-04-10)에서 CRITICAL #4로 식별.

## 결정
비루프백 바인딩 시 TLS 설정을 강제한다:
1. bind_addr가 127.0.0.1 또는 ::1이 아니면, `[tls] enabled=true` 필수
2. 조건 미충족 시 gRPC 서버 시작 거부 + 에러 로그
3. 기존 127.0.0.1 기본값은 유지 (대부분 환경에서 변경 불필요)
4. TLS가 활성이면 REST와 동일한 인증서 재사용 (daemon.conf [tls] cert/key)

## 결과
- 좋음: 평문 토큰 네트워크 노출 원천 차단
- 좋음: 기존 127.0.0.1 사용자에게 영향 없음
- 나쁨: 멀티노드 gRPC 사용 시 TLS 인증서 필수 (배포 복잡성 증가)
- 포기한 것: gRPC에 별도 인증서를 사용 — 관리 부담, REST TLS 재사용으로 충분

## 하지 않기로 한 것
- gRPC를 완전히 제거하지 않는다. 내부 통신 경로가 필요할 때 REST보다 효율적인 선택지로 남긴다.
- mTLS(상호 인증)를 구현하지 않는다. 현재 토큰 + TLS 조합으로 충분.
